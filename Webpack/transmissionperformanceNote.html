Transmission Performance 传输性能

在优化传输性能时要考虑到以下方面:

  • 总传输量: 所有需要传输的 JS 文件的内容加起来，就是总传输量。重复代码越少，总传输量越少。
  • 文件数量: 当访问页面时，需要传输的 JS 文件数量。文件数量越多，http 请求越多，响应速度越慢。
  • 浏览器缓存: JS 文件会被浏览器缓存，被缓存的文件不会再进行传输。

分包

通过提取大量的公共代码，将整体的代码，分布到不同的打包文件中。简单说就是从已有的 chunk 中分离出新的 chunk。从而减少公共代码以降低总体积，并且可以充分利用浏览器缓存，提高传输与构建性能。

当多个 chunk 引入了公共模块，并且公共模块的体积较大或较少改动时，例如一些大型的第三方库，可以进行分包处理。

手动分包

先单独打包公共模块，公共模块会被打包成为 dll (Dynamic Link Library) 动态链接库，并生成资源清单。

打包公共模块

然后根据入口模块进行正常打包，打包时如果发现模块中使用了资源清单中描述的模块，则打包结果中会根据资源清单直接导出相应模块的全局变量，也就是说在打包结果中不会包含这些模块的代码。

手动分包的过程:

  1. 打包公共模块，暴露变量名并生成资源清单。

    • 新建用于公共模块打包的 webpack 配置文件，一般命名为 webpack.dll.config.js。

    • 编写 webpack 配置文件，通常配置如下:

      // webpack.dll.config.js

const webpack = require("webpack");
const path = require("path");

module.exports = {
    mode: "production", // 一般不会调试公共模块，以生产模式打包。
    entry: {
        jquery: ["jquery"], // 后续交由 DllPlugin 处理，需要配置为数组。
        lodash: ["lodash"] // 后续交由 DllPlugin 处理，需要配置为数组。
    },
    output: {
        filename: "dll/[name].js",
        library: "[name]" // 配置每个 bundle 暴露的全局变量名，以便在代码运行时引用。
    },
    plugins: [
        new webpack.DllPlugin({
            path: path.resolve(__dirname, "dll", "[name].manifest.json"), // 配置资源清单的保存位置。
            name: "[name]" // 配置资源清单中暴露的变量名。
        }) // 使用 webpack 自带插件 DllPlugin 生成资源清单。
    ]
};

    • 通过这个配置文件运行打包，即可完成公共模块打包。

  2. 在 HtmlWebpackPlugin 插件设置的页面模板中手动引入之前打包好的公共模块 JS 文件，例如:

    <script src="./dll/jquery.js"></script>
<script src="./dll/lodash.js"></script>

  3. 打包自己项目文件，控制打包结果。

    • 在打包过程中如果使用 clean-webpack-plugin 插件，为保证之前打包的公共模块不会清除，需要对它配置如下:

      new CleanWebpackPlugin({ // 配置要清除的文件或目录。
    cleanOnceBeforeBuildPatterns: ["**/*", '!dll', '!dll/*'] // 排除掉 dll 目录本身和它里面的文件。
})

      目录和文件的匹配规则使用的是 globbing patterns。

    • 使用 webpack 自带的 DllReferencePlugin 插件控制打包结果，以使用资源清单中之前打包好的相应公共模块，其配置如下:

      module.exports = {
    plugins: [
        new webpack.DllReferencePlugin({
            manifest: require("./dll/jquery.manifest.json")
        }),
        new webpack.DllReferencePlugin({
            manifest: require("./dll/lodash.manifest.json")
        })
    ]
}

    • 通过 webpack 指令完成自己项目文件打包。

手动分包的注意事项:

  • 资源清单不参与运行，可以不放到打包目录中。
  • 必须手动引入公共模块 JS 文件，以及避免被删除。
  • 不要对小型的公共 JS 库使用，因为会增加文件请求数量。

手动分包的优点:

  • 极大提升自身模块的打包速度。
  • 极大的缩小了自身文件体积。
  • 有利于浏览器缓存第三方库的公共代码。

手动分包的缺点:

  • 使用非常繁琐。
  • 如果第三方库中包含重复代码，则效果不太理想。

自动分包

不同于手动分包，自动分包是从实际的角度出发，从一个更加宏观的角度来控制分包，一般不对具体哪个包要分出去进行控制。因此使用自动分包，不仅非常方便，而且更加贴合实际的开发需要。

自动分包流程

从自动分包流程中可以看出，分包策略至关重要，它决定了如何分包，分包时 webpack 开启了一个新的 chunk，对分离的模块进行打包，打包结果中，公共的部分被提取出来形成了一个单独的文件，它是新 chunk 的产物。

自动分包策略的配置:

要控制自动分包，关键是要配置一个合理的分包策略，有了分包策略之后，不需要额外安装任何插件，webpack 会自动的按照策略进行分包。

实际上，webpack 在内部是使用自带的插件 SplitChunksPlugin 进行分包的。

自动分包策略需要通过 webpack 的 optimization.splitChunks 选项配置。事实上，分包策略有其默认的配置，我们只需要轻微的改动，即可应对大部分分包场景。

  • chunks: 该配置项用于配置需要应用分包策略的 chunk，一般配置为 all。
    它有三个取值，分别是:

    • all: 【默认】对于所有的 chunk 都要应用分包策略。
    • async: 仅针对异步 chunk 应用分包策略。
    • initial: 仅针对普通 chunk 应用分包策略。

  • maxSize: 该配置可以控制包的最大字节数。如果某个包 (包括分出来的包) 超过了该值，则 webpack 会尽可能的将其分离成多个包，默认值是 0。
    但要注意，分包的基础单位是模块，如果一个完整的模块超过了该体积，它是无法做到再切割的。因此，尽管使用了这个配置，完全有可能某个包还是会超过这个体积。
    另外，虽然该配置可以把一些包进行再切分，但是实际的总体积和传输量并没有发生变化，所以实际意义其实不大。

    如果要进一步减少公共模块的体积，可以进行压缩和 tree-shaking。

  • automaticNameDelimiter: 该配置可以控制新 chunk 名称的分隔符，默认值是 ~。

  • minChunks: 该配置可以控制一个模块被多少个 chunk 使用时，才会进行分包，默认值是 1。

  • minSize: 该配置项用于配置当分包达到多少字节后才允许被真正的拆分，默认值是 20000。

  • cacheGroups: 该配置项用于配置缓存组，默认值如下:

    cacheGroups: {
    // 配置的属性名是缓存组名称，会影响到分包的 chunk 名。
    // 配置的属性值对象是缓存组的配置，缓存组继承所有的全局配置，也有自己特殊的配置。
    defaultVendors: {
        test: /[\\/]node_modules[\\/]/, // 当匹配到相应模块时，将这些模块进行单独打包。
        priority: -10 // 缓存组优先级，优先级越高该策略越先进行处理。
    },
    default: {
        minChunks: 2, // 覆盖全局配置，将最小 chunk 引用数改为 2。
        priority: -20,
        reuseExistingChunk: true // 如果 chunk 已经从主包中分离，重用已经被分离出去的 chunk。
    }
}

    之前配置的分包策略是全局的，而实际的分包策略是基于缓存组的。
    每个缓存组提供一套独有的策略，webpack 按照缓存组的优先级依次处理每个缓存组，被缓存组处理过的分包不需要再次分包。
    缓存组可以继承和/或覆盖 splitChunks.* 中的任何选项。但 test、 priority 和 reuseExistingChunk 只能在缓存组级别配置。若要禁用任何默认缓存组，请将其设置为 false。

// 利用缓存组来完成对公共样式的抽离

module.exports = {
    optimization: {
        splitChunks: {
            chunks: "all",
            cacheGroups: {
                styles: {
                    test: /\.css$/, // 匹配样式模块。
                    minSize: 0, // 覆盖默认的最小尺寸，这里仅作测试。
                    minChunks: 2 // 覆盖默认的最小 chunk 引用数。
                }
            }
        }
    },
    module: {
        rules: [{
            test: /\.css$/,
            use: [MiniCssExtractPlugin.loader, "css-loader"]
        }]
    },
    plugins: [
        new CleanWebpackPlugin(),
        new HtmlWebpackPlugin({
            template: "./public/index.html",
            chunks: ["index"]
        }),
        new MiniCssExtractPlugin({
            filename: "[name].[hash:5].css",
            chunkFilename: "common.[hash:5].css" // 配置来自于分割 chunk 的文件名。
        })
    ]
}

自动分包的注意事项:

在多页应用中，如果使用 html-webpack-plugin 插件生成相应的页面，它会自动的找到相应入口文件中分离出去的 chunk，并在生成的页面中引用，插件配置如下:

new HtmlWebpackPlugin({
    template: "./public/index.html",
    chunks: ["index"] // 配置需要生成页面的入口文件。
})

自动分包的原理:

  1. 检查每个 chunk 编译的结果。
  2. 根据分包策略，找到那些满足策略的模块。
  3. 根据分包策略，生成新的 chunk 打包这些模块。
    在分包的代码中，会加入一个全局变量，类型为数组，其中包含公共模块的代码。
  4. 把打包出去的模块从原始包中移除，并修正原始包代码。
    在原始包的代码中，使用全局变量中的公共代码。

代码压缩

在生产环境中使用代码压缩，可以减少代码体积，同时破坏代码的可读性 (提升破解成本)。

webpack 内置了代码压缩工具 terser，terser 支持 ES6+ 语法，当使用生产环境打包时即会用其进行代码压缩。

通过 terser 代码压缩可以将代码中的换行或注释等删除，将没有使用到的代码、无法访问到的代码剔除，将变量名等简化，将判断语句转换为三目运算符，将常量的运算直接转换为运算结果，将仅调用一次的函数转换为处理结果，但会保留有副作用的函数。[1]另外 terser 还提供一些参数配置对代码压缩进行控制，可以在其官网查看: https://terser.org/

测试 terser 对代码的压缩效果: https://try.terser.org/

如果需要更改、添加别的压缩工具，又或者需要对 terser 进行配置，使用下面的 webpack 配置即可。

const TerserPlugin = require('terser-webpack-plugin');
const OptimizeCSSAssetsPlugin = require('optimize-css-assets-webpack-plugin');

module.exports = {
    optimization: {
        minimize: true, // 是否要启用压缩，默认情况下无需设置，生产环境会自动开启。
        minimizer: [ // 压缩时使用的插件，可以有多个，默认情况下在生产环境使用 terser 代码压缩。
            new TerserPlugin(),
            new OptimizeCSSAssetsPlugin()
        ]
    }
}

使用 terser-webpack-plugin 可对 JS 代码压缩，CSS 代码压缩需要使用 optimize-css-assets-webpack-plugin。使用前需要先安装对应的包。

Tree Shaking

代码压缩可以移除模块内部的无效代码，而 Tree Shaking 可以移除模块之间的无效代码。

有些模块导出的代码并不一定会全被用到，特别是一些大型的第三方库，例如 lodash 。而 Tree Shaking 就是用于在生产环境中移除掉不会用到的导出。

如需使用 Tree Shaking 优化代码，只需配置为生产环境即可，在生产环境中 Tree Shaking 会自动开启。

Tree Shaking 只支持 ES6 模块化语法，因为 ES6 模块化导入导出具有以下特点:

  • 导入导出语句只能是顶层语句。
  • import 的模块名只能是字符串常量。
  • import 绑定的变量是不可变的。

这些特征都非常有利于分析出稳定的依赖，webpack 在具体分析依赖时，会先保证代码正常运行，然后再尽量使用 Tree Shaking。所以，如果依赖的是一个导出的对象，由于 JS 语言的动态特性，以及 webpack 还不够智能，为了保证代码正常运行，它不会移除对象中的任何信息。因此，在编写代码的时候尽量使用 export xxx 导出，而不使用 export default {xxx} 导出，使用 import {xxx} from "xxx" 导入，而不使用 import xxx from "xxx" 导入。

依赖分析完毕后，webpack 会根据每个模块每个导出判断是否被使用，将未使用的导出标记为 dead code，然后交给代码压缩工具处理，而代码压缩工具最终移除掉那些标记为 dead code 的代码。

对于使用 CommonJS 模块化语法的模块，比如 lodash 等一些第三方库，是无法使用 Tree Shaking 的。因此，要寻找这些库的 ES6 版本，比如 lodash-es。同时要确保没有编译器将 ES6 模块语法转换成 CommonJS 模块语法，比如 Babel 预设 @babel/preset-env 的默认行为。

Tree Shaking 本身并没有完善的作用域分析，可能导致在一些标记了 dead code 的导出函数中的依赖仍然会被视为依赖。而插件 webpack-deep-scope-plugin 提供了作用域分析，它可以使 webpack 消除与未使用的导出相关的未使用的导入，从而解决这个问题，但是这个插件缺少维护。

Tree Shaking 在移除无效代码的同时，会保留具有副作用的导出函数。[1:1]当 webpack 无法确定某个模块是否有副作用时，为保证代码正确运行，它往往将其视为有副作用，例如下面的情景:

// common.js
var n  = Math.random();

// index.js
import "./common.js"

虽然没有用 common.js 的导出，但 webpack 担心 common.js 有副作用，如果去掉会影响某些功能，所以打包后的文件中任然会出现 common.js 中的代码。对此有两种解决方法:

  • 在语句级别中通过使用 /*#__PURE__*/ 注释告诉 webpack 这个函数调用是无副作用的。
  • 在模块级别中通过在 package.json 中使用 sideEffects 属性标记该文件是没有副作用的，通常是一些第三方库在它们自己的 package.json 中标注，它有两种配置方式:
    • false: 当前工程中所有模块都没有副作用。注意，这种写法会影响到某些 CSS 文件的导入。
    • 数组: 设置哪些文件拥有副作用，例如: ["!src/common.js"]，表示只要不是 src/common.js 文件都有副作用。

类似于 JS 文件的 Tree Shaking，对于 CSS 文件，因为不是 ES6 模块化语法，所以需要通过插件 purgecss-webpack-plugin 实现，但这个插件不能使用 Css Module。

懒加载

懒加载会先把代码在一些逻辑断点处分离开，等到在一些代码块中完成某些操作后，立即引用或即将引用另外一些新的代码块。这样加快了应用的初始加载速度，减轻了它的总体体积，因为某些代码块可能需要等到使用的时候再加载或者永远不会被加载。

懒加载作为异步的代码块，会在自动分包时会打包成一个新的文件 (需要针对异步 chunk 应用分包策略)。当需要动态加载时浏览器会使用 JSOP 的方式远程去读取一个 JS 模块，并返回一个 promise 对象。

懒加载在异步代码中通过 import() 语句实现，它可以指定生成的 chunk 名，并在后面指定需要动态加载的模块，例如:

const btn = document.querySelector("button");
btn.onclick = async function() {
  const { chunk } = await import(/* webpackChunkName:"lodash" */"lodash-es");
  const result = chunk([3, 5, 6, 7, 87], 2);
  console.log(result);
};

由于 import() 语句并不是正式的 ES6 语法，所以无法直接使用 Tree Shaking，不过可以通过导入静态文件的方式解决，例如:

// index.js
const { chunk } = await import("./util");

// util.js
export { chunk } from "lodash-es";

  1. 副作用 (side effect)
    在函数运行过程中，可能会对外部环境造成影响的功能。如果函数中包含异步代码、localStorage、对外部数据修改的代码，该函数就是副作用函数。如果一个函数没有副作用，同时，函数的返回结果仅依赖参数，则该函数叫做纯函数 (pure function)。 ↩︎ ↩︎
