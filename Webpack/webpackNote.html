webpack笔记

webpack CLI 打包文件

webpack <entry> <output>

为了更合适且方便地使用配置，可以在 webpack.config.js 中对 webpack 进行配置，但也可以通过 CLI 参数 --config 来指定某个配置文件。

为了让不同的环境有自己特定的配置，可以通过 CLI 参数 --env 来指定环境名称，然后在配置文件中进行判断，根据环境名称返回不同的配置结果。

webpack --env abc # env: "abc"
webpack --env.abc # env: {abc:true}
webpack --env.abc=1  # env： {abc:1}
webpack --env.abc=1 --env.bcd=2 # env: {abc:1, bcd:2}

CLI 中传入的任何参数会在配置文件中映射为对应的参数。当命令行参数与配置文件中的配置出现冲突时，以命令行参数为准。

webpack 配置文件

配置文件中通过 CommonJS 模块导出一个对象，对象中的各种属性对应不同的 webpack 配置。
需要注意，配置文件中的代码，必须是有效的 node 代码

module.exports = {
    mode: XXX,
    entry: XXX,
    ...
}

配置文件中可以针对多个环境分别书写 webpack 配置，允许通过 CommonJS 模块导出一个函数。在开始构建时，webpack 发现配置是一个函数，会调用该函数，将函数返回的对象作为配置内容。

函数接收一个参数，这个参数的值来自于 CLI 中 webpack 命令的 --env 参数指定的值，因此可以根据不同的环境返回不同的对象。

var baseConfig = require("./webpack.base");
var devConfig = require("./webpack.dev");
var proConfig = require("./webpack.pro");

module.exports = function (env) {
    if (env && env.prod) {
        return {
            ...baseConfig,
            ...proConfig
        }
    }
    else {
        return {
            ...baseConfig,
            ...devConfig
        }
    }
}

webpack 中的 loader 模块文件

loader 模块文件中通过 CommonJS 模块导出一个函数，它的作用是将某个源码字符串转换成另一个源码字符串返回，简单说就是转换代码格式。

函数接收一个参数，这个参数为交由 loader 处理的模块的源码字符串。

函数返回 loader 处理过的源码字符串。

需要注意，loader 模块文件中的代码，必须是有效的 node 代码

函数可通过 loader-utils 第三方库的 getOptions 方法，获取 this 上下文对象在配置文件中传递的参数。

var loaderUtils = require("loader-utils");

module.exports = function(sourceCode){
    var options = loaderUtils.getOptions(this);
    var reg = new RegExp(options.changeVar, "g");
    return sourceCode.replace(reg, "var");
}

webpack 中的 plugin 模块文件

plugin 模块文件中通过 CommonJS 模块导出一个类函数，其本质就是一个带有 apply 方法的对象。它的作用是以各种方式自定义 webpack 构建过程，简单说就是把功能嵌入到 webpack 的编译流程中。

类函数中有一个 apply 方法，并向该方法中传入一个 compiler 对象，这个对象提供了大量的钩子函数（hooks，可以理解为事件），通过注册这些事件并指定事件类型创建事件处理函数。

事件处理函数中传入一个 compilation 对象，同样这个对象也提供了大量的钩子函数，用相同的方法可以创建 compilation 事件处理函数。

类函数可通过自己的构造函数，获取在配置文件中传递的参数。

module.exports = class MyPlugin{
  
    constructor(XXX = XXX){
        this.XXX = XXX;
    }

    apply(compiler){
        compiler.hooks.事件名称.事件类型("MyPlugin", function(compilation){
            console.log(this.XXX); //事件处理函数
            compilation.hooks.事件名称.事件类型("MyPlugin", 事件处理函数)
        })
    }
}

事件名称

即要监听的事件名，也就是钩子函数的名称，分为 compiler 钩子和 compilation 钩子。

事件类型

这里使用的是 Tapable API ，这个小型的库是一个专门用于钩子函数监听的库，它提供了一些事件类型：

  • tap：注册一个同步的钩子函数，函数运行完毕则表示事件处理结束。
  • tapAsync：注册一个基于回调的异步的钩子函数，函数通过调用一个回调表示事件处理结束。
  • tapPromise：注册一个基于 Promise 的异步的钩子函数，函数通过返回的 Promise 进入已决状态表示事件处理结束。

事件处理函数
参与 webpack 编译和生成。

需要注意，plugin 模块文件中的代码，必须是有效的 node 代码

webpack 配置选项

  • CLI <entry> | CONF entry {string/object}:
    指定入口文件的相对路径，从这个起点开始，应用程序启动执行。如果传递一个数组，那么数组的每一项都会执行。多入口文件在配置文件中以对象形式书写，在 CLI 中以键值对形式书写。
    默认值：./src/index.js
  • CLI <output> | CONF output {string/object}:
    指定编译结果文件。
    默认值：./dist/main.js

    • CLI --output-path | CONF path {string}:
      资源放置的文件夹，必须配置一个绝对路径。
      默认值：path.resolve(__dirname, "dist")

    • CLI --output-filename | CONF filename {string}:
      针对资源列表的文件名或路径的配置，可以是包含以下替换模板字符串的动态名称用于解决缓存问题。

      模板 描述
      [hash] 模块标识符(module identifier)的 hash
      [chunkhash] chunk 内容的 hash
      [name] 模块名称
      [id] 模块标识符(module identifier)

      [hash] 和 [chunkhash] 的长度可以使用 [hash:16]（默认为20）来指定。
      [id] 在开发环境下就是模块名称，在生产环境下是数字，不推荐作为唯一标识符。

      默认值：main.js

    • CLI --output-library | CONF library {string}:
      以库的形式导出入口文件，也就是将打包结果中自执行函数的执行结果暴露给指定的变量。与 DllPlugin 和 DllReferencePlugin 插件一起使用，以提高传输性能。[1]

    • CLI --output-library-target | CONF libraryTarget {string (var/window/this/global/commonjs/...)}:
      以库的形式导出入口文件时的输出的类型，也就是控制如何暴露入口包的导出结果。
      默认值：var

    • CONF publicPath {string}:
      当按需加载或加载外部资源时，用来确定最终在浏览器中所引用的正确的路径。该选项的值作为运行或载入时所创建的每个 URL 的前缀。在多数情况下，此选项的值都会以 / 结束。并且该选项的值可以在入口文件中使用 __webpack_public_path__ 来获取，以便在运行时动态读取或设置。webpack-dev-server 也会默认以 publicPath 为基准，来决定在哪个目录下启用服务，来访问 webpack 输出的文件。
      默认值：空

  • CLI --config {string}:
    指定配置文件。
    默认值：webpack.config.js
  • CLI --mode | CONF mode {string (production/development)}:
    指定编译结果代码运行的环境，会影响 webpack 对编译结果代码格式的处理。
    默认值：production
  • CLI --devtool | CONF devtool {string (none/eval/source-map/...)}:
    为打包好的资源定义 source map 的类型。[2]
    默认值：pro=none / dev=eval
  • CONF module {object}:
    针对模块的配置。
    • CONF noParse {reg}:
      不解析正则表达式匹配的模块，通常用它来忽略那些大型的单模块库或没有任何依赖的已经打包好的模块，以提高构建性能。[3]
    • CONF rules {array}:
      模块匹配规则，可以存在多个规则。
      • Rule {object}:
        每个规则是一个对象。
        • CONF test {reg}:
          匹配模块的规则。
        • CONF use {array}:
          匹配到后应用的 loader 模块。
          • UseEntry {string/object}:
            其中一个 loader 模块或 loader 模块的路径。
            • CONF loader {string}:
              指定 loader 模块的相对路径，该字符串会被放置到 require 中。
            • CONF options {string/object}:
              向对应 loader 模块传递的额外参数。也可以在 loader 配置中以 query 的形式传参。
        • CONF include {reg}:
          仅包含需要应用 loader 的场景，用于提高构建性能。[3:1]
        • CONF exclude {reg}:
          排除需要应用 loader 的场景，用于提高构建性能。[3:2]
  • CLI --plugin | CONF plugins {array}:
    加载 plugin 模块。
  • CONF context {string}:
    用于从配置中解析入口文件和 loader 模块，入口和 loader 的相对路径会以 context 的配置作为基准路径，必须配置一个绝对路径。
    默认值：当前目录
  • CLI --target | CONF target {string (web/node/...)}:
    设置打包结果最终要运行的环境。
    默认值：web
  • CONF resolve {object}:
    用于控制模块解析过程。
    • CONF modules {array}:
      解析模块遇到导入语句时应该搜索的目录。绝对路径和相对路径都能使用，webpack 会从数组中配置的位置依次从当前目录以及祖先路径寻找依赖的模块。
      默认值："node_modules"
    • CLI --resolve-extensions | CONF extensions {array}:
      自动解析确定的扩展。当解析模块遇到无具体后缀的导入语句时，例如 require("test") ，会按照数组中的配置依次查找与后缀名匹配的文件。
      默认值：".js", ".json"
    • CLI --resolve-alias | CONF alias {object}:
      指定模块的别名。导入导出语句中可以加入配置的键名，来确保模块引入变得更简单，例如 require("@/abc.js") 。在大型系统中，源码结构往往比较深和复杂，别名配置可以让我们更加方便的导入依赖。
  • CONF externals {object}:
    从最终的 bundle 中排除掉依赖的源码，而是在运行时再去从外部获取这些扩展依赖。适用于一些第三方库来自于外部 CDN 的情况，可以让 bundle 的体积变得更小，还不影响源码的运行。
  • CONF stats {string/object}:
    控制构建过程中控制台的输出内容。webpack 提供一些预设选项，也可以更加精细的控制，了解更多。
  • CLI --watch 或 -w | CONF watch {boolean}:
    用于观察依赖文件的变化，一旦有变化，则可以重新执行构建流程。在初始构建之后，webpack 将继续监听任何已解析文件的更改。
    默认值：false
  • CONF devServer {object}:
    配置 webpack-dev-server 。
    • CLI webpack-dev-server --port | CONF port {number}:
      指定要监听请求的端口号。
    • CLI webpack-dev-server --open | CONF open {boolean}:
      运行 devServer 时是否自动打开浏览器。
    • CLI webpack-dev-server --open-page | CONF openPage {string}:
      指定打开浏览器时要导航到的页面路径。
    • CLI --hot | CONF hot {boolean}:
      启用 webpack 的模块热替换特性，用于提高构建性能。[3:3]
    • CONF index {string}:
      指定索引文件的文件名。通过域名和端口号就可直接访问。
    • CONF proxy {object}:
      配置代理，用于跨域访问。会将匹配到的 URL 字符串替换为代理地址后再发送请求，此过程发生在 node 环境中。因 dev-server 使用了 http-proxy-middleware 包，proxy 的配置可参考其文档。[4]
    • CONF stats {string/object}:
      配置控制台显示的 bundle 信息，了解更多。
  • CONF optimization {object}:
    用于优化性能。
    • CONF splitChunks {object}:
      用于实现自动分包，内部使用 webpack 自带的 SplitChunksPlugin 插件，它提供一套默认配置作为 web 性能的最佳实践，用于提高传输性能。[1:1]
      • CONF chunks {string (all/async/initial)}:
        配置需要应用分包策略的 chunk。
        默认值：all
      • CONF maxSize {number}:
        用于将大于所设置字节的 chunk 分割为更小的包。
        默认值：0
      • CONF automaticNameDelimiter {string}:
        配置生成 chunk 名称的分隔符。
        默认值：~
      • CONF minChunks {number}:
        配置模块被多少个 chunk 使用时才会进行分包。
        默认值：1
      • CONF minSize {number}:
        配置当分包达到多少字节后才允许被真正的拆分。
        默认值：20000
      • CONF cacheGroups {object}:
        用于配置缓存组。可以继承和/或覆盖 splitChunks 中的任何选项。
        默认值：defaultVendors, default
        • Name {object/false}:
          配置缓存组的名称，会影响到分包的 chunk 名。若要禁用任何默认缓存组，请将其设置为 false。
          • CONF test {reg}:
            控制此缓存组匹配哪些模块。
            默认值：[\/]node_modules[\/]
          • CONF priority {number}:
            配置缓存组优先级。
            默认值：-10 和 -20
          • CONF reuseExistingChunk {boolean}:
            如果 chunk 已经从主包中分离，配置是否重用已经被分离出去的 chunk。
            默认值：true
    • CLI --optimize-minimize | CONF minimize {boolean}:
      配置是否使用 TerserPlugin 或 optimization.minimizer 中指定的插件来压缩包。
      默认值：true (仅生产环境中）
    • CONF minimizer {array}:
      配置压缩时使用的插件，可以有多个。
      默认值：new TerserPlugin() (仅生产环境中）

webpack 模块化兼容性

由于 webpack 同时支持 CommonJS 和 ES6 module 模块化

  • 同模块化标准

    如果导出和导入使用的是同一种模块化标准，打包后的效果和之前学习的模块化没有任何差异。

  • 不同模块化标准

    不同的模块化标准，webpack 按照如下的方式处理。

webpack 编译过程

webpack 的作用是将源代码编译（构建、打包）成最终代码。

整个过程大致分为三个步骤

  1. 初始化
  2. 编译
  3. 输出

初始化

此阶段，webpack 会将 CLI 参数、配置文件、默认配置进行融合，形成一个最终的配置对象，为接下来的编译阶段做必要的准备。

对配置的处理过程是依托一个第三方库 yargs 完成的。

编译

  1. 创建 chunk

    chunk 是 webpack 在内部构建过程中的一个概念，译为“块”，它表示通过某个入口找到的所有依赖的统称。

    根据入口模块（默认为 ./src/index.js ）创建一个 chunk 。

    每个 chunk 都有至少两个属性：
    name：名字，默认为 main 。
    id：唯一编号，开发环境和 name 相同，生产环境是一个从 0 开始的数字。

  2. 构建所有依赖模块

    通过入口模块构建所有依赖模块，完成后 chunk 中会产生一个模块列表，列表中包含了模块id和模块转换后的代码。

    在读取文件内容后，会根据配置文件中配置的模块的匹配规则，将匹配到的文件的内容交由相应的 loaders 模块进行处理，再把处理好的文件做抽象语法树分析。如果配置文件中没有配置的模块的匹配规则，可以理解为跳过处理 loaders 的步骤。

    loaders 模块会根据配置文件中的模块匹配规则读取相应的 loader ，并将这些 loader 放入一个数组中，在处理时会将数组中的 loader 从后向前执行，生成最后处理好的文件。

  3. 产生 chunk assets

    通过模块列表，webpack 会根据初始化形成的最终配置为 chunk 生成一个资源列表，即 chunk assets ，资源列表可以理解为是生成到最终文件的文件名和文件内容。

    每个资源列表都有一个 chunk hash ，chunk hash 是根据所有 chunk assets 的内容生成的一个 hash 字符串。[5]

  4. 合并 chunk assets

    将多个 chunk assets 合并到一起产生总的 assets ，并产生一个总的 hash 。

输出

webpack 将利用 node 中的 fs模块（文件处理模块），根据编译产生的总的 assets ，生成相应的文件。

涉及术语

  • module：模块，分割的代码单元，webpack 中的模块可以是任何内容的文件，不仅限于 JS 。
  • chunk：webpack 内部构建模块的块，一个 chunk 中包含多个模块，这些模块是从入口模块通过依赖分析得来的。
  • bundle：chunk 构建好模块后会生成 chunk 的资源清单，清单中的每一项就是一个 bundle ，可以认为 bundle 就是最终生成的文件。
  • hash：最终的资源清单所有内容联合生成的 hash 值。
  • chunkhash：chunk 生成的资源清单内容联合生成的 hash 值。
  • chunkname：chunk 的名称，如果没有配置则使用 main 。
  • id：通常指 chunk 的唯一编号，如果在开发环境下构建和 chunkname 相同，如果是生产环境下构建，则使用一个从0开始的数字进行编号。

webpack 插件

插件的作用是以各种方式自定义 webpack 构建过程，简单说就是把功能嵌入到 webpack 的编译流程中。

webpack 会在初始化阶段构建 compiler 对象，创建好 Compiler 对象后，会运行 plugin 模块中的 apply 函数，根据 compiler 对象上注册的监听事件以及事件类型触发相应的事件处理函数。

随后 webpack 会在编译阶段前构建 compilation 对象，创建好 compilation 对象后，根据 compilation 对象上注册的监听事件以及事件类型触发相应的事件处理函数。

在整个 webpack 打包期间只有一个 compiler 对象，随后完成打包工作的是 compiler 对象内部创建的 compilation 对象。当文件发生变化，需要重新打包时，会重新创建 compilation 对象。

webpack-dev-server

webpack-dev-server 可以在开发阶段提供了一个简单的 web 服务器来模拟真实环境，并且能够实时重新加载以避免开发过程中打包、运行、调试过程过于繁琐。

安装: npm install --save-dev webpack-dev-server

使用: webpack-dev-server

webpack-dev-server 支持大多数的 webpack CLI 命令参数。

webpack 有着丰富的插件接口，webpack 自身的多数功能都使用这个插件接口。所有的 webpack 内置插件都作为 webpack 的静态属性存在。

    DOMAIN: JSON.stringify("duyi.com") // DOMAIN = "duyi.com"
})

这样在源码中，就可以直接使用插件中提供的常量，当 webpack 编译完成后，会自动替换为常量的值。

BannerPlugin

它可以为每个 chunk 生成的文件头部添加一行注释，一般用于添加作者、公司、版权等信息，例如：

new webpack.BannerPlugin({
    banner: `
    hash:[hash]
    chunkhash:[chunkhash]
    name:[name]
    author:yuanjin
    corporation:duyi
    `
})

ProvidePlugin

自动加载模块，而不必到处 import 或 require 。当源码中用到相应的 identifier 时 module 就会自动被加载，并且 identifier 会被这个 module 输出的内容所赋值。

new webpack.ProvidePlugin({
    $: 'jquery', // identifier: 'module1'
    _: 'lodash'
})

HotModuleReplacementPlugin

用于开发阶段启用热模块替换(HMR)，当配置 devServer.hot 的值为 true 时会自动添加使用该插件，无需手动引用，它用于提高构建性能。[3:4]

new webpack.HotModuleReplacementPlugin({
     // 大多数情况下不需要配置任何选项。
})

DllPlugin

这个插件会生一个名为 manifest.json 的文件作为资源清单。与 output.library 的选项相结合可以暴露出 dll 函数，并与 DllReferencePlugin 插件配合可实现分包功能，以提高传输性能。[1:2]

new webpack.DllPlugin({
    context: __dirname,
    name: "[name]_[hash]",
    path: path.join(__dirname, "manifest.json")
})

DllReferencePlugin

这个插件把只有 dll 的 bundle 引用到需要的预编译的依赖。与 DllPlugin 插件配合可实现分包功能，以提高传输性能。[1:3]

new webpack.DllReferencePlugin({
    context: __dirname,
    manifest: require("./manifest.json"),
    name: "./my-dll.js",
    scope: "xyz",
    sourceType: "commonjs2"
})

SplitChunksPlugin

用于实现自动分包，通过配置 optimization.splitChunks 直接使用该插件，它提供一套默认配置作为 web 性能的最佳实践，它用于提高传输性能。[1:4]

    这里的构建性能指的是在开发阶段的构建性能，并且只会影响开发效率。主要目的是降低从打包开始到代码效果呈现所经过的时间。构建性能越高，开发过程中时间的浪费越少。 ↩︎ ↩︎ ↩︎ ↩︎ ↩︎
